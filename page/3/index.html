Flume

Flume

高可用、高可靠、分布式的日志的采集、聚合和传输系统。

组件

  1. Agent
    1. Java进程
    2. 由Source、Channel和Sink组成
  2. Source
    1. 收集数据
    2. 支持多种源格式的数据
    3. 把日志封装为Event发送给Channel
  3. Channel
    1. Source和Sink的缓冲区
    2. MemoryChannel和FileChannel两种
  4. Sink
    1. 把数据写入存储系统
    2. 支持多种输出
    3. 轮询Channel中的数据
  5. Event
    1. 数据的封装结果
    2. Header(K-V结构)和Body(字节数组)

处理流程

  1. Source接收数据
  2. Channel Processor处理
  3. Interceptor链进行拦截和过滤
  4. ChannelSelector进行分发
    1. Replicating把Event分发到所有的Channel
    2. Multiplexing把Event分发到不同的Channel
  5. Event通过不同的Channel分发到不同Sink处理
  6. SinkProcessor处理
    1. DefaultSinkProcessor
      1. 输出到单个Sink
    2. LoadBalancingSinkProcessor
      1. 多个Sink进行负载均衡
      2. 两种均衡算法
        1. 随机分配
        2. 轮询
    3. FailoverSinkProcessor
      1. 多个Sink进行错误恢复
      2. 定义优先级并且优先选择优先级高的Sink
  7. Sink把数据输出

Source

  1. ExecSource
    1. 监控日志文件并且实时增加
    2. 不支持断点续传
    3. 使用tail -f读取文件, 不记录读取位置
  2. TailDirSource
    1. 支持断点续传
    2. 监控多个文件夹
    3. 记录文件inode，offset，file_name来判断读取的文件和位置

MySQL

Join的底层实现

MySQL Join的实现原理是NestLoop

Simple Nested-Loop

驱动表中的每一个记录和被驱动表中的记录进行比较

Index Nested-Loop

驱动表使用索引，被驱动表和驱动表比较时使用索引进行加速

Block Nested-Loop Join

主从复制

复制过程

  1. binlog记录主数据库的更新
  2. 主通过log dump线程写binlog
  3. 从开启IO线程拉取binlog到relay log中
  4. 从开启SQL线程对relay log进行重放，实现主从复制

复制机制

  1. binlog记录模式
  2. GTID复制模式

复制模式

根据主数据库是否会在执行事务的过程中主动向从数据库推送binlog分为不同模式

  1. 异步模式
    1. 主数据库通过异步的dump线程向从数据库发送binlog
    2. 执行的线程不管复制是否成功就commit
  2. 半同步模式
    1. 主数据库需要收到从数据库的同步结果才commit
    2. 主数据库接受到一个从节点的ack后就commit
  3. 同步模式
    1. 主数据库接受到所有从节点的ack后才commit

锁

这里的锁不是分布式锁

是否锁住资源

  1. 悲观锁
    1. 访问数据时加锁，不允许其他线程访问
    2. synchronized
    3. Lock类
  2. 乐观锁
    1. 访问数据时不加锁
    2. 判断数据没被修改则执行操作
    3. 如果数据被修改则执行其他操作
    4. CAS
      1. CompareAndSwap
      2. 操作系统的cmpxchg指令

竞争失败是否阻塞

  1. 自旋锁

synchronized的锁升级

synchronized的多种形态，对应着不同的锁竞争水平

  1. 无锁
    1. 程序中不会有锁的竞争，只有主线程？
  2. 偏向锁
    1. 程序中会出现锁的竞争，但是只有1个线程来加锁
    2. 当出现另一个线程来竞争，则升级为轻量级锁
  3. 轻量级锁
    1. 自旋锁
      1. 资源被占用时，线程不阻塞，而是在自己的CPU时间内重复尝试获取锁
  4. 重量级锁
    1. 当线程竞争轻量级锁比较激烈时，升级为重量级锁。
    2. 竞争资源的线程发现重量级锁时，放弃自旋并直接挂起

锁是否是公平竞争

  1. 公平锁
  2. 非公平锁

锁是否可以共享

  1. 共享锁
  2. 独占锁
